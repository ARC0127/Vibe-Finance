# 定时设计

所有时间均为 Asia/Shanghai。平台自动化负责唤醒研究任务，Python 流水线负责确定性校验和账本更新。

## 短周期

- 计划：周一至周五 16:30。
- 先核验交易所日历；非交易日只记录休市，不伪造行情或订单。
- 若当日不可变报告已经存在，读取并核验，禁止覆盖。
- 从 A/B 级来源优先采集，C 级只交叉验证。
- 创建 data/inbox/YYYY-MM-DD.json，先运行 validate，通过后运行 --mode short。
- 任何数据缺失、冲突、陈旧或市场冲击都 fail closed。
- 只生成虚拟的下一开盘订单，不连接真实交易。

## 长周期

- 计划：每周日 20:00。
- 汇总本周日报、订单、持仓、费用、来源失败和基准。
- 只在已结束样本上审查策略；样本不足时不得改规则。
- 运行 --mode long 或生成等价周报，已有同日长报时不得覆盖。

## 活动监测

- 计划：每 6 小时。
- 运行 python3 -m vibe_finance status。
- 只读检查 data/ledger/heartbeat.json、最近决策报告和 API 预算。
- 监测任务本身禁止更新心跳、订单或账本。
- 常规阈值为 36 小时；考虑周末/法定休市时，以最近应运行的短周期和 8 天内的长周期为补充判断。
- 只有状态陈旧、文件缺失、预算异常或运行失败时才需要告警。

## 高频状态

当前为 DISABLED_NOT_CREATED。只有以下条件全部通过才允许创建：

1. 许可明确的逐笔/盘口点时数据；
2. 正确交易日历、公司行为和代码生命周期；
3. 排队、部分成交、涨跌停、停牌、滑点和费用模型；
4. 走前回测以及未用于调参的样本外测试；
5. 压力测试、资源与 API 成本预算；
6. 明确的启停、熔断和回滚规则；
7. 用户再次确认。

“高频定时”不是高频交易能力；没有上述条件时增加调用频率只会放大噪声和成本。
